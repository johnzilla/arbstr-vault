**1. Create src/modules/approvals/approvals.repo.ts:**

Import `pendingApprovals` from schema, `eq`, `and`, `lte` from drizzle-orm. Use the narrow Db type (`BetterSQLite3Database<Record<string, never>>`).

Methods:

- `create(db, values: { agent_id, type, transaction_id, amount_msat, destination?, expires_at: Date })` — inserts a pending_approvals row, returns the inserted row via `.returning().get()`.

- `findById(db, id: string)` — `.where(eq(pendingApprovals.id, id)).get()`. Returns row or undefined.

- `findByTransactionId(db, transactionId: string)` — `.where(eq(pendingApprovals.transaction_id, transactionId)).get()`. Used by payment-status route.

- `claimForResolution(db, id: string, newStatus: 'approved' | 'denied' | 'timed_out')` — Atomic CAS update. `db.update(pendingApprovals).set({ status: newStatus, resolved_at: new Date() }).where(and(eq(pendingApprovals.id, id), eq(pendingApprovals.status, 'pending'))).returning().get()`. Returns the updated row if CAS succeeded, undefined if already resolved. This prevents the race condition between manual resolution and timeout checker (Research Pitfall 3).

- `findExpired(db)` — `db.select().from(pendingApprovals).where(and(eq(pendingApprovals.status, 'pending'), lte(pendingApprovals.expires_at, new Date()))).all()`. Returns array of expired pending approvals.

- `countPending(db, agentId: string)` — returns count of pending approvals for an agent. Used by dashboard (Plan 03).

- `listPending(db)` — returns all pending approvals (used by operator dashboard).

**2. Create src/modules/approvals/approvals.service.ts:**

Import approvalsRepo, ledgerRepo, auditRepo, webhookService. Accept `db` as parameter (not import) for testability.

Methods:

- `createApproval(db, { agentId, type, txId, amountMsat, destination?, expiresAt })` — wraps in IMMEDIATE transaction:
  1. Write RESERVE ledger entry: `ledgerRepo.insert(tdb, { id: txId, agent_id: agentId, amount_msat: -amountMsat, entry_type: 'RESERVE', ref_id: txId, mode: 'simulated' })`
  2. Insert pending_approvals row: `approvalsRepo.create(tdb, { agent_id: agentId, type, transaction_id: txId, amount_msat: amountMsat, destination, expires_at: expiresAt })`
  3. Write APPROVAL_REQUESTED audit entry
  - After transaction commits, fire webhook (non-blocking): `webhookService.send({ event: 'approval_required', agent_id: agentId, transaction_id: txId, amount_msat: amountMsat }).catch(() => {})`
  - Return the approval row.

  CRITICAL: The RESERVE and pending_approvals row MUST be in the SAME transaction. If they split, a crash between them leaves balance reserved with no approval record (Research anti-pattern).

- `resolveApproval(db, approvalId: string, resolution: 'approved' | 'denied')` — uses CAS:
  1. `const claimed = approvalsRepo.claimForResolution(db, approvalId, resolution)`
  2. If !claimed, return null (already resolved — caller returns 409)
  3. If `resolution === 'denied'`:
     - IMMEDIATE transaction: write RELEASE (+amount_msat), write APPROVAL_DENIED audit
     - Fire `approval_denied` webhook (non-blocking)
     - Return `{ ...claimed, payment_status: 'FAILED' }`
  4. If `resolution === 'approved'`:
     - IMMEDIATE transaction: write APPROVAL_GRANTED audit
     - Return `{ ...claimed, payment_status: 'APPROVED' }`
     - Note: The ACTUAL wallet payment execution for approved payments is handled by the approve route handler (it calls the wallet, then writes settlement/failure). This keeps the service focused on approval state management.

- `expireTimedOut(db)` — called by setInterval in index.ts:
  1. `const expired = approvalsRepo.findExpired(db)`
  2. For each: `approvalsRepo.claimForResolution(db, approval.id, 'timed_out')`
  3. If claimed: IMMEDIATE transaction write RELEASE + APPROVAL_TIMEOUT audit
  4. Fire `approval_timeout` webhook (non-blocking)
