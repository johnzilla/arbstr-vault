**In src/modules/payments/payments.service.ts:**

Add a new `executeApprovedPayment` method to the object returned by `createPaymentsService`. This method handles executing a wallet payment after an operator approves a pending approval. It must be added alongside the existing `processPayment` method inside the returned object from `createPaymentsService`.

The method signature:
```typescript
async executeApprovedPayment(
  db: DB,
  agentId: string,
  transactionId: string,
  amountMsat: number,
  destination: string | null,
): Promise<{ status: 'SETTLED' | 'FAILED'; payment_hash?: string; fee_msat?: number }>
```

Implementation logic:
1. The RESERVE ledger entry already exists (written when REQUIRE_HUMAN_APPROVAL was triggered). Do NOT write another RESERVE.
2. Build a wallet request using `simulatedWallet` (the simulated wallet is always available). For the initial implementation, use the simulated wallet since the approval was created under the simulated mode. The wallet request needs: `amount_msat`, `asset: 'BTC_simulated'`, `purpose: 'approved_payment'`, `destination_type: 'internal'`, `destination: destination ?? 'approved'`, `transaction_id: transactionId`, `max_fee_msat: 1000`.
3. Call `options.simulatedWallet.pay(walletReq)` to execute the payment.
4. On SETTLED result:
   - Write an IMMEDIATE transaction that:
     - Inserts a PAYMENT ledger entry: `{ agent_id: agentId, amount_msat: -amountMsat, entry_type: 'PAYMENT', ref_id: transactionId, mode: 'simulated' }`. Do NOT set `id: transactionId` because the RESERVE already has that id. Let drizzle auto-generate a new id.
     - Writes a RELEASE ledger entry: `{ agent_id: agentId, amount_msat: amountMsat, entry_type: 'RELEASE', ref_id: transactionId, mode: 'simulated' }` to cancel the original RESERVE.
     - Writes a PAYMENT_SETTLED audit entry: `{ agent_id: agentId, action: 'PAYMENT_SETTLED', amount_msat: amountMsat, ref_id: transactionId, metadata: { approval_execution: true } }`
   - After the transaction, call `alertsService.checkAndNotify(db as unknown as Db, agentId).catch(() => {})` for post-settlement balance alert (OBSV-06).
   - Return `{ status: 'SETTLED' }`.

5. On FAILED result (or caught exception):
   - Write an IMMEDIATE transaction that:
     - Inserts a RELEASE ledger entry: `{ agent_id: agentId, amount_msat: amountMsat, entry_type: 'RELEASE', ref_id: transactionId, mode: 'simulated' }` to restore the reserved balance.
     - Writes a PAYMENT_FAILED audit entry: `{ agent_id: agentId, action: 'PAYMENT_FAILED', amount_msat: amountMsat, ref_id: transactionId, metadata: { approval_execution: true, error: 'wallet_failed' } }`
   - Return `{ status: 'FAILED' }`.

6. Wrap the entire method body in try/catch. On any unexpected error, write RELEASE + PAYMENT_FAILED audit and return FAILED (fail-closed).

IMPORTANT: The net ledger effect for a settled approved payment is:
  - RESERVE: -amount_msat (already written when entering PENDING_APPROVAL)
  - RELEASE: +amount_msat (cancels the RESERVE)
  - PAYMENT: -amount_msat (the actual payment debit)
  Net: -amount_msat (correct — the agent spent the payment amount)

This RESERVE+RELEASE+PAYMENT pattern avoids trying to "upgrade" the existing RESERVE entry type, which would be fragile. The RELEASE cancels the hold, and the PAYMENT is the actual settlement debit. This mirrors the pattern used in the deny path (RELEASE only, net: 0) and timeout path (RELEASE only, net: 0).
