**The `pay()` method implementation:**

1. Import `subscribeToPayViaRequest` from `lightning` (handle ESM/CJS same way as lnd.client.ts in Plan 01).

2. Return a Promise that wraps the EventEmitter subscription:

```
const sub = subscribeToPayViaRequest({
  lnd: this.lnd,
  request: req.destination,      // BOLT11 invoice string
  max_fee_mtokens: String(maxFeeMsat),  // fee limit from policy, passed via PaymentRequest
});
```

3. **Event handlers:**

   - `sub.on('paying', ({ id }) => { ... })` — Payment in-flight, `id` is the payment_hash hex string. Store this in memory synchronously (DO NOT await a DB write here — regtest can emit 'confirmed' synchronously after 'paying'). The payment_hash is returned as part of the PaymentResult.

   - `sub.on('confirmed', ({ id, fee_mtokens }) => { ... })` — Payment succeeded. Resolve the promise with `{ transaction_id, status: 'SETTLED', mode: 'lightning', payment_hash: id, fee_msat: Number(fee_mtokens), settled_at: new Date() }`.

   - `sub.on('failed', ({ is_route_not_found, is_pathfinding_timeout, is_insufficient_balance }) => { ... })` — Payment definitively failed. Resolve the promise with `{ transaction_id, status: 'FAILED', mode: 'lightning', payment_hash: paymentHashFromPayingEvent }`. Include failure reason in the result for audit logging.

   - `sub.on('error', (err) => { ... })` — gRPC stream error. This is NOT a payment failure (see Pitfall 1 in RESEARCH.md). Reject the promise with a custom `LightningStreamError` class so the caller can distinguish stream errors from payment failures. The caller (payment service) must keep the ledger in PENDING state on stream error.

4. **PaymentRequest extension:** The `pay()` method needs `max_fee_msat` from the policy. Two options:
   - Option A: Add `max_fee_msat?: number` to the wallet's PaymentRequest interface
   - Option B: Pass fee limit via the constructor or a separate config
   Use Option A — add `max_fee_msat?: number` to PaymentRequest in wallet.interface.ts. The payment service reads it from the policy and passes it through.

5. **Export:** `export { LightningWallet }` and also `export class LightningStreamError extends Error { ... }` for the stream error type.

**CRITICAL anti-patterns to avoid (from RESEARCH.md):**
- DO NOT treat `sub.on('error')` as payment failure — keep PENDING
- DO NOT use `payViaPaymentRequest` (blocks, no 'paying' event for payment_hash capture)
- DO NOT do async DB write in 'paying' handler — store payment_hash in memory first (Pitfall 2)

**Modify the processPayment flow for RESERVE/RELEASE pattern (per locked decisions):**

The current two-phase transaction flow changes:

**Phase 1 (sync IMMEDIATE tx):** Same as before — read balance/daily_spend, evaluate policy, write PAYMENT_REQUEST audit. If ALLOW, also read `max_fee_msat` from the policy for the wallet call.

**Between Phase 1 and Phase 2 (async wallet call):**
- For Lightning mode ONLY: write a RESERVE ledger entry (-amount_msat, entry_type='RESERVE', mode='lightning', ref_id=txId) in a quick sync transaction BEFORE calling the wallet. This debits the balance immediately.
- For simulated mode: keep existing behavior (no RESERVE needed — settles instantly).

**Wallet call:** `await wallet.pay({ ...request, transaction_id: txId, max_fee_msat: policyMaxFee })`

**Phase 2 — On wallet result:**
- If `status === 'SETTLED'` (Lightning or simulated):
  - For Lightning: write PAYMENT_SETTLED audit entry + FEE debit ledger entry if fee_msat > 0 (a separate negative ledger entry for the routing fee, entry_type='PAYMENT', ref_id=txId, mode='lightning'). The RESERVE already debited the principal.
  - For simulated: keep existing behavior (write PAYMENT ledger debit + PAYMENT_SETTLED audit atomically)

- If `status === 'FAILED'` (Lightning only — simulated never fails):
  - Write RELEASE ledger entry (+amount_msat, entry_type='RELEASE', mode='lightning', ref_id=txId) to credit back the reserved amount
  - Write PAYMENT_FAILED audit entry with failure reason

- If wallet throws `LightningStreamError` (gRPC disconnect):
  - Do NOT write RELEASE — payment may still be in-flight
  - Write PAYMENT_REQUEST audit already happened in Phase 1
  - Return response with status='PENDING'
  - The payment_hash was captured by LightningWallet's 'paying' handler — update the RESERVE ledger entry's payment_hash column if available (for crash recovery)
