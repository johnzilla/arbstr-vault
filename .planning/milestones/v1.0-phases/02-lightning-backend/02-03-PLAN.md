**Create `src/routes/agent/payment-status.routes.ts`:**

Implement `GET /agents/:id/payments/:tx_id` — per locked decision in CONTEXT.md.

Route handler:
1. Apply `agentAuth` hook (same pattern as other agent routes)
2. Validate params: `id` starts with `ag_`, `tx_id` starts with `tx_`
3. Query the ledger for all entries with `ref_id = tx_id` and `agent_id = id`
4. From the entries, determine payment status:
   - If any entry has `entry_type = 'RELEASE'` -> status is FAILED
   - If any entry has `entry_type = 'PAYMENT'` (for simulated) or there's a RESERVE with no RELEASE -> check further
   - If there's a RESERVE and no RELEASE and no PAYMENT -> status is PENDING
   - If there's a PAYMENT entry (simulated mode) or (RESERVE and no RELEASE for lightning settled) -> status is SETTLED
   - Actually simpler: look at entry types present for this ref_id:
     - Has RELEASE -> FAILED (refunded)
     - Has only RESERVE (no RELEASE, no additional PAYMENT for fee) -> PENDING
     - Has RESERVE and the RESERVE was settled (need a marker) OR has PAYMENT (simulated) -> SETTLED

   NOTE: Plan 02 writes a PAYMENT_SETTLED audit entry on settlement. For ledger, Lightning settled payments have a RESERVE entry (and optionally a FEE PAYMENT entry). Simulated payments have a PAYMENT entry. Use the audit_log to determine status more cleanly:

   Query approach: Check audit_log for the ref_id:
   - Has `PAYMENT_SETTLED` action -> SETTLED
   - Has `PAYMENT_FAILED` action -> FAILED
   - Has only `PAYMENT_REQUEST` -> PENDING (if Lightning mode) or should not happen for simulated

   OR add a `getPaymentStatus` method to ledgerRepo that queries ledger entries by ref_id and determines status from entry types present.

5. Response schema (per CONTEXT.md locked decision):
```json
{
  "transaction_id": "tx_01...",
  "payment_hash": "abc123...",
  "status": "PENDING" | "SETTLED" | "FAILED",
  "mode": "lightning" | "simulated",
  "amount_msat": 10000,
  "fee_msat": 100,
  "created_at": "2026-02-26T...",
  "settled_at": "2026-02-26T..."
}
```

6. If no entries found for the tx_id, return 404.
