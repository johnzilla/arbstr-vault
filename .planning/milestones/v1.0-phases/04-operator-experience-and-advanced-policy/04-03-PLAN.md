**1. Create src/routes/agent/withdrawals.routes.ts:**

Fastify plugin with ZodTypeProvider. Apply `agentAuth` hook.

Route: `POST /agents/:id/withdrawals`

Zod schemas:
- params: `{ id: z.string().startsWith('ag_') }`
- body: `{ destination: z.string().min(1), amount_msat: z.number().int().positive() }`
  - `destination` is a BOLT11 invoice string
  - `amount_msat` is the withdrawal amount in millisatoshis
- response 200: `{ withdrawal_id: z.string(), transaction_id: z.string(), status: z.literal('PENDING_APPROVAL'), amount_msat: z.number() }`
- response 400: `{ error: { code: z.string(), message: z.string() } }`
- response 403: same error shape

Handler logic:
1. Generate txId via `generateTransactionId()`
2. Read agent from `agentsRepo.findById(db, id)` — return 404 if not found
3. Read current policy from `policyVersionsRepo.getCurrent(db, id)` — if null, return 403 with 'no_policy_configured'
4. **Run policy evaluation** (Research Pitfall 6): Call `evaluatePolicy(policy, { balance_msat, daily_spent_msat, request_amount_msat: amount_msat })`. If outcome is DENY, return 400 with reason. Do NOT skip policy check for withdrawals.
5. **Withdrawals always require approval** (locked decision). Regardless of evaluatePolicy outcome (even if ALLOW), create a pending approval:
   - Calculate expiresAt from `policy.approval_timeout_ms ?? 300_000`
   - IMMEDIATE transaction:
     a. Write RESERVE: `ledgerRepo.insert(tdb, { id: txId, agent_id: id, amount_msat: -amount_msat, entry_type: 'RESERVE', ref_id: txId, mode: 'simulated' })`
     b. Insert pending_approvals: `approvalsRepo.create(tdb, { agent_id: id, type: 'withdrawal', transaction_id: txId, amount_msat, destination, expires_at: expiresAt })`
     c. Write WITHDRAWAL_REQUESTED audit entry with metadata including destination
6. Fire webhook (non-blocking): `webhookService.send({ event: 'withdrawal_requested', agent_id: id, transaction_id: txId, amount_msat }).catch(() => {})`
7. Return 200: `{ withdrawal_id: approval.id, transaction_id: txId, status: 'PENDING_APPROVAL', amount_msat }`

NOTE: When the operator approves a withdrawal, the approve route (from Plan 02) calls the wallet to pay the BOLT11 invoice. The withdrawal reuses the existing Lightning payment flow — the `destination` field on the pending_approvals row is the BOLT11 invoice.

**2. Create src/modules/alerts/alerts.service.ts:**

Balance alert service with per-agent cooldown tracking.

```typescript
import { webhookService } from '../webhook/webhook.service.js';
import { policyVersionsRepo } from '../agents/agents.repo.js';
import { ledgerRepo } from '../ledger/ledger.repo.js';
import { auditRepo } from '../audit/audit.repo.js';

// In-memory cooldown tracker: agentId -> last alert timestamp
const lastAlertTime = new Map<string, number>();

export const alertsService = {
  /**
   * Check if an agent's balance has dropped below their configured alert_floor_msat.
   * If below floor and cooldown has elapsed, fire a balance_alert webhook.
   *
   * MUST only be called after a payment SETTLES (not after RESERVE).
   * Research Pitfall 4: firing alert after RESERVE produces spurious alerts for payments that fail.
   */
  async checkAndNotify(db: any, agentId: string): Promise<void> {
    const policy = policyVersionsRepo.getCurrent(db, agentId);
    if (!policy?.alert_floor_msat || policy.alert_floor_msat <= 0) return;

    const balance = ledgerRepo.getBalance(db, agentId);
    if (balance >= policy.alert_floor_msat) return;

    // Check cooldown
    const cooldownMs = policy.alert_cooldown_ms ?? 3_600_000;
    const lastAlert = lastAlertTime.get(agentId) ?? 0;
    if (Date.now() - lastAlert < cooldownMs) return;

    // Fire alert
    lastAlertTime.set(agentId, Date.now());

    // Write BALANCE_ALERT audit entry
    auditRepo.insert(db, {
      agent_id: agentId,
      action: 'BALANCE_ALERT',
      metadata: {
        balance_msat: balance,
        alert_floor_msat: policy.alert_floor_msat,
      },
    });

    // Webhook — non-blocking
    webhookService.send({
      event: 'balance_alert',
      agent_id: agentId,
      balance_msat: balance,
      alert_floor_msat: policy.alert_floor_msat,
    }).catch(() => {});
  },

  /** Reset cooldown for testing */
  _resetCooldowns(): void {
    lastAlertTime.clear();
  },
};
```

**1. Create src/routes/admin/dashboard.routes.ts:**

Fastify plugin with ZodTypeProvider. Apply `adminAuth` hook.

Route: `GET /operator/dashboard`

Zod schemas:
- querystring: `{ status: z.enum(['active', 'all']).default('active'), sort_by: z.enum(['balance', 'daily_spend', 'name']).default('name'), sort_order: z.enum(['asc', 'desc']).default('asc') }`
- response 200:
```typescript
z.object({
  agents: z.array(z.object({
    agent_id: z.string(),
    name: z.string(),
    balance_msat: z.number(),
    daily_spend_msat: z.number(),
    daily_utilization_pct: z.number(),
    pending_approvals_count: z.number(),
    policy: z.object({
      version: z.number(),
      max_transaction_msat: z.number(),
      daily_limit_msat: z.number(),
      max_fee_msat: z.number().nullable(),
      approval_timeout_ms: z.number().nullable(),
      alert_floor_msat: z.number().nullable(),
      alert_cooldown_ms: z.number().nullable(),
    }).nullable(),
    last_payment_at: z.string().nullable(),
    balance_below_floor: z.boolean(),
    created_at: z.string(),
  })),
  total_agents: z.number(),
  total_pending_approvals: z.number(),
})
```

Handler implementation:
1. Load all agents via `agentsRepo.list(db, { limit: 1000 })` (personal-use scale, no pagination needed per locked decision)
2. For each agent, aggregate:
   - `balance` = `ledgerRepo.getBalance(db as Db, agent.id)`
   - `dailySpend` = `ledgerRepo.getDailySpend(db as Db, agent.id)`
   - `policy` = `policyVersionsRepo.getCurrent(db as Db, agent.id)`
   - `pendingCount` = `approvalsRepo.countPending(db as Db, agent.id)`
   - `lastPayment` = query audit_log for the most recent PAYMENT_SETTLED or PAYMENT_FAILED entry for this agent (use `auditRepo.listByAgent` with limit 1 and appropriate action filter, or add a helper method)
   - `dailyUtilization` = policy?.daily_limit_msat ? Math.round((dailySpend / policy.daily_limit_msat) * 100) : 0
   - `belowFloor` = policy?.alert_floor_msat ? balance < policy.alert_floor_msat : false
3. Sort by the requested sort_by field and sort_order
4. Compute totals: `total_agents`, `total_pending_approvals` (sum of all agents' pending counts)
5. Return the assembled response

At personal-use scale (10-50 agents), N+1 queries are fine — each call is sub-millisecond on SQLite. The Research doc confirms this design choice.

For `last_payment_at`: Query audit_log where agent_id = agent.id AND action IN ('PAYMENT_SETTLED', 'PAYMENT_FAILED') ordered by created_at DESC limit 1. Extract created_at as ISO string. If no payments found, return null. This could be a new helper `auditRepo.getLastPaymentTimestamp(db, agentId)` or done inline.
